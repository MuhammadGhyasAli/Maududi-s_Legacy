%% ================== EXECUTIVE SUMMARY ===================
\section{Executive Summary}

Maududi's Legacy is a full-stack web application that preserves and provides intelligent access to 77 works of Sayyid Abul A'la Maududi (1903--1979), one of the most influential Islamic scholars of the 20th century. As both the senior developer and AI developer on this project, I designed the system around a core architectural principle: \textbf{AI-powered interaction must be central to the user experience while maintaining scholarly accuracy}. This drove the development of a dual-provider AI chat system---Google Gemini 2.0 Flash as primary with automatic failover to Groq-hosted Llama models---that enables book-specific Q\&A (chat with PDFs), cross-book semantic search (AI Context Finder), image-based queries, and multi-language support across six languages including RTL scripts.

The platform is deployed with a Next.js 15 frontend (React 19, TypeScript, Tailwind CSS v4, Redux Toolkit) on Vercel and a Python FastAPI backend on PythonAnywhere. The application serves approximately 40,000--50,000 monthly API requests, achieving sub-20-millisecond response times for book data through aggressive caching (TTLCache + SWR pattern) and a local seed data fallback that circumvents the PythonAnywhere free plan's network restrictions.

\textbf{Key AI Contributions:}
\begin{itemize}[leftmargin=*,nosep]
    \item Designed the dual-provider LLM orchestrator with automatic failover (Gemini $\rightarrow$ Groq)
    \item Built the AI Context Finder --- a cross-book semantic search engine with structured citation extraction
    \item Implemented book-specific "chat with PDFs" functionality enabling focused Q\&A on individual works
    \item Developed an accuracy enforcement system with temperature-controlled generation and prompt-level constraints
    \item Created per-book AI personas via unique \texttt{aiContext} prompts for all 77 works
\end{itemize}

\subsection{Chat-Related Components}

The AI chat interface is built from four key components:

\begin{enumerate}
    \item \textbf{ChatPage.tsx} --- Orchestrates the per-book chat view. Loads the book's \texttt{aiContext} from the API, manages conversation state, injects language and accuracy instructions into each user message.
    \item \textbf{AiContextFinderModal.tsx} --- Implements the global search interface. On mount, fetches all 77 book titles and dynamically constructs a system prompt listing every work as the AI's knowledge base. Supports both text and image input.
    \item \textbf{ChatMessageList.tsx} --- Renders the conversation history. Contains a regex-based \texttt{parseStructuredResponse()} function that extracts citations in the format \texttt{**Book Title:**, **Chapter/Section:**, **Page Number:**, **Context:**} from AI responses.     \item \textbf{ChatInputArea.tsx} --- Input field with file attachment support. Handles base64 image encoding and language selector dropdown.
\end{enumerate}

\subsection{API Service Layer}

The \texttt{apiService.ts} module wraps all backend communication with:

\begin{itemize}
    \item \textbf{SWR caching}: 30-minute TTL with a 5-minute stale window; data served instantly from cache while background revalidation occurs.
    \item \textbf{Request deduplication}: In-flight requests for the same resource are coalesced into a single network call.
    \item \textbf{AbortController integration}: Pending requests are cancelled when the component unmounts or navigates away.
    \item \textbf{Base URL resolution}: Uses \texttt{NEXT\_PUBLIC\_API\_BASE\_URL} environment variable, defaulting to \texttt{http://localhost:8000} in development server and empty string (same-origin proxy) in production.
\end{itemize}

\subsection{Seed Data Fallback Pattern}

A critical architectural decision is the automatic fallback from MongoDB to local seed data. When MongoDB is unreachable (as it is on the PythonAnywhere free plan, which blocks all outbound TCP connections), the books router transparently serves data from \texttt{data/books.py}:

\begin{lstlisting}[language=Python, caption=Seed data fallback in books router]
async def _seed_books() -> list[dict]:
    return BOOKS_SEED_DATA  # 77 books from data/books.py

@router.get("", response_model=list[BookResponse])
async def get_books(category: str | None = None, db=Depends(get_db)):
    if db is None:
        books = await _seed_books()
        return _filter_by_category(books, category)
    # ... MongoDB path ...
\end{lstlisting}

The \texttt{database.py} module caches MongoDB failure in a \texttt{\_mongodb\_failed} global flag. After the first unsuccessful connection attempt (2-second timeout), subsequent requests bypass \texttt{MongoClient} creation entirely, reducing response time from approximately 1.4~seconds to approximately 13~milliseconds. This was one of the critical optimizations I implemented to ensure the app remained responsive on the PythonAnywhere free plan, which blocks all outbound TCP connections.

\subsection{Middleware Chain}

Six middleware layers execute in sequence for each request:

\begin{enumerate}
    \item \textbf{CORS} (\texttt{CORSMiddleware}) --- Allows origins from \texttt{settings.cors\_origins}; added to default FastAPI middleware stack.
    \item \textbf{Request body size limit} --- Custom middleware returning 413 Payload Too Large for bodies exceeding 10~MB (critical for preventing large image uploads to non-chat endpoints).
    \item \textbf{Security headers} --- \texttt{X-Content-Type-Options: nosniff}, \texttt{X-Frame-Options: DENY}, \texttt{X-XSS-Protection: 1}, Strict-Transport-Security, and \texttt{Referrer-Policy: strict-origin-when-cross-origin}.
    \item \textbf{Structured logging} --- \texttt{BaseHTTPMiddleware} wrapping \texttt{logging\_middleware}, logging request method, path, status, and duration via \texttt{structlog}.
    \item \textbf{Exception handlers} --- Global and request-specific handlers set CORS headers on error responses (addressing a known \texttt{CORSMiddleware} limitation where it does not intercept exception handler responses).
    \item \textbf{Rate limiting} --- \texttt{slowapi.Limiter} applied per-route with token bucket semantics and IP-based keying.
\end{enumerate}

%=================== AI FEATURES (CORE) ===================
\section{AI Features in Detail}
\label{sec:ai-features}

As the AI developer on this project, my primary architectural decision was to treat the AI layer as a critical infrastructure component that must never have a single point of failure. This led to the dual-provider design with automatic failover, accuracy enforcement at every layer, and per-book AI personas that constrain the model's knowledge scope. The following subsections detail each AI subsystem.

\subsection{Dual-Provider Architecture}

The AI subsystem's central component is the \texttt{LLMService} orchestrator, which implements a failover chain between two independent AI providers. I designed this as a simple loop over a provider list rather than a complex routing system — simplicity was essential for reliability on the constrained PythonAnywhere platform:

\begin{lstlisting}[language=Python, caption=LLMService orchestrator with automatic failover]
class LLMService:
    def generate_response(
        self,
        system_instruction: str,
        messages: list[dict]
    ) -> str:
        enhanced = (system_instruction or "") + ACCURACY_INSTRUCTION

        for name, service in [
            ("Gemini", gemini_service),
            ("Groq", groq_service)
        ]:
            try:
                result = service.chat(enhanced, messages)
                if result.get("error"):
                    continue
                return result["response"]
            except Exception:
                continue

        raise RuntimeError("All AI providers failed")
\end{lstlisting}

The provider order is deliberate: Gemini is attempted first because it handles both text and images natively without model switching, offers lower latency, and has more generous free-tier quotas. Groq acts as the safety net, with model auto-selection based on content type.

\subsubsection{Google Gemini Service}

\begin{itemize}
    \item \textbf{Model}: \texttt{gemini-2.0-flash}
    \item \textbf{SDK}: \texttt{google.generativeai} (lazy-imported on first call)
    \item \textbf{System instruction}: Passed via \texttt{GenerativeModel} constructor
    \item \textbf{Generation config}: \texttt{temperature=0.1}, \texttt{top\_p=0.9}, \texttt{max\_output\_tokens=4096}
    \item \textbf{History}: Reconstructs turn-based conversation from flat message list
    \item \textbf{Failure mode}: Catches all exceptions (network errors, API errors, quota exhaustion) and returns an error dict for the orchestrator to handle
\end{itemize}

\subsubsection{Groq Service}

\begin{itemize}
    \item \textbf{Text model}: \texttt{llama-3.3-70b-versatile}
    \item \textbf{Vision model}: \texttt{llama-3.2-11b-vision-preview} (auto-selected when messages contain \texttt{image\_url} parts)
    \item \textbf{SDK}: \texttt{groq} Python package
    \item \textbf{Role normalization}: Maps \texttt{ai} and \texttt{model} roles to \texttt{assistant} for Groq API compatibility
    \item \textbf{System instruction}: Prepended as a message with role \texttt{system}
    \item \textbf{Generation config}: \texttt{temperature=0.1}, \texttt{max\_tokens=4096}, \texttt{top\_p=0.9}
    \item \textbf{Graceful degradation}: Service initializes without error even when \texttt{GROQ\_API\_KEY} is not configured
\end{itemize}

\subsection{Accuracy Enforcement System}

Every AI request, regardless of provider or mode, is augmented with the following accuracy instruction:

\begin{lstlisting}[language=Python, caption=Accuracy instruction appended to all system prompts]
ACCURACY_INSTRUCTION = (
    "\n\nCRITICAL ACCURACY RULES:\n"
    "1. Only state facts you are highly confident about."
    " Never fabricate or guess.\n"
    "2. If you are unsure about a detail,"
    " explicitly say so instead of speculating.\n"
    "3. Cite exact book titles and specific references when available.\n"
    "4. A concise, correct answer is far better"
    " than a lengthy speculative one.\n"
    "5. Quality over quantity -- prioritize factual precision"
    " over comprehensiveness."
)
\end{lstlisting}

This is concatenated with the book-specific or global system prompt in \texttt{LLMService.generate\_response()}. The instruction is reinforced by the low temperature setting (0.1), which minimizes the model's creative variance and biases it toward deterministic, factual outputs.

\subsection{Per-Book Chat}

\subsubsection{Purpose and Workflow}

The per-book chat feature enables users to engage in a focused Q\&A session constrained to a single work. Each of the 77 books has a unique \texttt{aiContext} string (stored in \texttt{data/books.py}) that serves as the system prompt, defining the AI's persona and scope.

Representative \texttt{aiContext} examples:

\begin{itemize}
    \item ``You are an AI trained on the contents of 'Tafheem ul Quran' by Sayyid Abul A'la Maududi. This is a comprehensive commentary of the Qur'an. Your purpose is to provide detailed explanations of Qur'anic verses based on Maududi's commentary, context, and interpretations.''
    \item ``You are an AI expert on 'Khutabat' by Sayyid Abul A'la Maududi. This collection comprises his Friday sermons covering various aspects of Islam. Answer questions based on the sermons' content and context.''
    \item ``You are an AI assistant specialized in 'Purdah' by Sayyid Abul A'la Maududi. This book explores the concept of Hijab and Purdah in Islam. Ground all answers in Maududi's arguments and references.''
\end{itemize}

\subsubsection{API Implementation}

\begin{lstlisting}[language=Python, caption=Per-book chat endpoint (actual implementation)]
@router.post("", response_model=ChatResponse)
@limiter.limit("20/minute")
async def chat(
    request: Request,
    chat_request: ChatRequest,
    db: Database = Depends(get_db)
):
    validate_book_id(chat_request.bookId)

    # Retrieve book via repository (MongoDB or fallback)
    book_repo = BookRepository(db)
    book_model = book_repo.get_book_by_id(chat_request.bookId)
    if not book_model:
        raise NotFoundException(
            f"Book with id {chat_request.bookId} not found")
    book = book_repo.to_pydantic(book_model)

    # Extract messages
    messages = [
        {"role": msg.role, "content": msg.content}
        for msg in chat_request.messages
    ]

    # Delegate to AI orchestrator with book's aiContext
    try:
        response = llm_service.generate_response(
            book.aiContext, messages)
        return ChatResponse(response=response)
    except RuntimeError as e:
        raise HTTPException(status_code=502, detail=str(e))
\end{lstlisting}

The flow proceeds as follows:

\begin{enumerate}
    \item Frontend sends \texttt{POST /api/v1/chat} with \texttt{\{bookId: int, messages: array\}}
    \item Backend validates \texttt{bookId} and retrieves the book from MongoDB or seed data
    \item The book's \texttt{aiContext} field is extracted as the system instruction
    \item \texttt{LLMService.generate\_response()} enhances it with the accuracy rules
    \item The enhanced prompt and message history are sent to Gemini (primary) or Groq (fallback)
    \item The AI response is returned as \texttt{\{"response": string\}}
\end{enumerate}

\subsubsection{Frontend Prompt Enhancement}

Before sending, \texttt{ChatPage.tsx} prepends a precision instruction to each user message:

\begin{lstlisting}[language=typescript, caption=Frontend prompt enhancement]
const enhancedMessage = `
Provide a precise, well-researched answer based
strictly on the book's content.
Accuracy is critical - only state what you are certain of.
If you are unsure about any detail, explicitly say so
rather than speculating.
Your entire response must be in \${selectedLanguage}.
When citing, provide full exact book titles
and specific context references.

My question: "\${userText}"
`;
\end{lstlisting}

The \texttt{selectedLanguage} variable is one of six options: English, Turkish, Urdu, Arabic, Persian, or Bengali. For right-to-left languages (Urdu, Arabic, Persian), the frontend applies CSS classes \texttt{font-nastaaliq} and \texttt{font-arabic}.

\subsection{Global AI Context Finder}

\subsubsection{Purpose}

The AI Context Finder enables cross-book semantic search. It allows users to ask questions like ``What does Maududi say about democracy?'' or ``Find me passages on the concept of justice'' and receive structured answers citing specific books, chapters, and page references.

\subsubsection{Dynamic System Prompt Construction}

Unlike per-book chat, which uses a static \texttt{aiContext} from the database, the AI Context Finder dynamically builds its system instruction on the frontend at runtime:

\begin{lstlisting}[language=typescript, caption=Dynamic system prompt construction]
const systemPrompt = `
You are an AI-powered search engine and expert archivist
for all works of Maududi. Your knowledge base includes:

\${allBooks.map(b => '- "' + b.title + '" by ' + b.author).join('\\n')}

When asked a question:
1. FIRST: Locate and cite the exact source:
   **Book Title:** [exact title]
   **Chapter/Section:** [chapter or section]
   **Page Number:** [approximate page]
   **Context:** [verbatim quote or close paraphrase]
2. THEN: Provide a comprehensive explanation
3. NEVER fabricate facts or citations.
   If you cannot find the information, say so clearly.

CRITICAL: You must search across ALL books.
Do not assume the answer is in just one book.
`;
\end{lstlisting}

\subsubsection{Citation Parsing and Rendering}

When the AI response arrives, the frontend's \texttt{parseStructuredResponse()} function (in \texttt{ChatMessageList.tsx}) applies the following regex to extract structured citations:

\begin{lstlisting}[language=typescript, caption=Citation parsing regex]
const regex = /\\*\\*Book Title:\\*\\*\\s*(.+?)\\s*\\n
              \\*\\*Chapter\\/Section:\\*\\*\\s*(.+?)\\s*\\n
              \\*\\*Page Number:\\*\\*\\s*(.+?)\\s*\\n
              \\*\\*Context:\\*\\*\\s*(.+?)(?=\\n\\*\\*|$)/gis;
\end{lstlisting}

Matched citations are rendered in a distinct green ``\texttt{Context Found}'' card with the following properties:

\begin{itemize}
    \item \textbf{Book Title}: Renders as a clickable hyperlink navigating to \texttt{/[category]/[bookSlug]}
    \item \textbf{Chapter/Section}: Displayed in bold
    \item \textbf{Page Number}: Formatted with a page icon
    \item \textbf{Context}: Displayed as a blockquote
\end{itemize}

The non-citation portion of the response is rendered as a standard chat message below the citation card.

\subsubsection{Multi-Modal Image Support}

The AI Context Finder supports image upload via a file input in \texttt{ChatInputArea.tsx}. The image processing pipeline:

\begin{enumerate}
    \item File is read as a base64 data URL on the client side using \texttt{FileReader.readAsDataURL()}
    \item The data URL is embedded in the message as:
    \begin{verbatim}
    {
      "role": "user",
      "content": [
        { "type": "text", "text": "..." },
        { "type": "image_url",
          "image_url": { "url": "data:image/...;base64,..." } }
      ]
    }
    \end{verbatim}
    \item The backend receives the message. If a \texttt{image\_url} part is detected, the Groq service automatically switches to \texttt{llama-3.2-11b-vision-preview} (Gemini handles images natively without model switching).
    \item The system prompt mandates a two-step verification protocol:
    \begin{itemize}
        \item \textbf{Step 1}: The AI identifies the book and asks for user confirmation.
        \item \textbf{Step 2}: After confirmation, the AI provides detailed analysis.
    \end{itemize}
\end{enumerate}

%=================== AUTHENTICATION ===================
\section{Authentication System}

\subsection{JWT-Based Authentication}

The authentication system uses JSON Web Tokens with the HS256 algorithm:

\begin{itemize}
    \item \textbf{Token expiry}: 1440 minutes (24 hours), configurable via \texttt{JWT\_EXPIRATION\_MINUTES}
    \item \textbf{Password hashing}: bcrypt via \texttt{passlib.context.CryptContext}
    \item \textbf{Token storage}: \texttt{localStorage('auth\_token')} on the client
    \item \textbf{Auth context}: React \texttt{AuthContext} restores session on mount via \texttt{GET /api/v1/auth/me}
\end{itemize}

\subsection{Authentication Flow}

Three authentication methods are supported:

\begin{enumerate}
    \item \textbf{Username/password registration and login}: Credentials verified against bcrypt hash in MongoDB; JWT returned upon success.
    \item \textbf{Google OAuth 2.0}: Frontend redirects to Google's OAuth endpoint; Google redirects back with an \texttt{id\_token}; backend verifies the token using \texttt{google.oauth2.id\_token.verify\_oauth2\_token()} and creates or links the user account.
    \item \textbf{Password reset}: Token-based flow with SHA256 hashed reset tokens stored in MongoDB with configurable TTL.
\end{enumerate}

\subsection{API Security}

\text{The \texttt{GET /api/v1/auth/me} endpoint is protected by FastAPI's \texttt{HTTPBearer} dependency, which extracts and validates the JWT from the \texttt{Authorization: Bearer <token>} header. All other endpoints (books, chat, login) are unauthenticated to allow public browsing while reserving authenticated features for future personalization.}

\subsection{PythonAnywhere Limitations}

The PythonAnywhere free plan imposes significant constraints that shaped the backend architecture:

\begin{itemize}
    \item \textbf{No outbound internet access}: All external TCP connections (including MongoDB Atlas on port 27017, PyPI for package installation, and API calls to external services) are blocked. This is the reason for the local seed data fallback and the offline wheel distribution.
    \item \textbf{Cold starts}: The webapp process is terminated after approximately 60--120 seconds of inactivity. On the next request, the bootstrap script reinstalls 103 wheel files (~67 MB), resulting in a 30--40 second cold start time. The server runs Python 3.10 (as specified in \texttt{final\_bootstrap.py}) while local development uses Python 3.13.
    \item \textbf{No persistent storage}: The filesystem is ephemeral; every restart begins from the uploaded source distribution.
\end{itemize}

\subsection{Deployment Pipeline}

The deployment to PythonAnywhere is managed by a custom \texttt{deploy\_pa.py} script that:

\begin{enumerate}
    \item Archives the backend source (excluding \texttt{node\_modules}, wheels for other platforms, and Git artifacts)
    \item Uploads the archive to PythonAnywhere via SCP/API
    \item Overwrites the \texttt{.env} file with production configuration
    \item Triggers a webapp reload via the PythonAnywhere API (\texttt{v1/websites/\{username\}/pythonanywhere.com/reload})
\end{enumerate}

The bootstrap script (\texttt{final\_bootstrap.py}) runs on every webapp start and performs:
\begin{enumerate}
    \item Environment variable loading from \texttt{.env}
    \item Virtual environment creation or reuse at \texttt{/home/syedghyas/.virtualenvs/maududi-venv/}
    \item Wheel installation from \texttt{backend/wheels\_linux/} (103 \texttt{.whl} files)
    \item Uvicorn server launch on the Unix socket specified by the \texttt{\$DOMAIN\_SOCKET} environment variable
\end{enumerate}

\subsection{CORS Configuration}

Cross-Origin Resource Sharing is configured to allow requests from the Vercel frontend domain. The \texttt{middleware.py} module also adds CORS headers to exception handler responses---a critical fix because FastAPI's built-in \texttt{CORSMiddleware} does not intercept responses produced by \texttt{@app.exception\_handler} decorators. Without this fix, browser-originated requests encountering a server error would receive a CORS error before seeing the actual error response, manifesting as ``Backend Unreachable'' in the frontend.

%=================== CONCLUSION ===================
\section{Conclusion \& Future Roadmap}

\subsection{Key Achievements}
 
The Maududi's Legacy project delivers a production-grade AI-powered digital archive with the following accomplishments:
 
\begin{itemize}
    \item \textbf{77 digitized works} with per-book AI context prompts, enabling focused, expert-level Q\&A on each text (chat with PDFs functionality).
    \item \textbf{Dual AI provider architecture} with automatic failover (Gemini $\rightarrow$ Groq), achieving high availability without a single point of failure for AI interactions.
    \item \textbf{Global semantic search} across all 77 books with structured citation extraction and clickable references (AI Context Finder functionality).
    \item \textbf{Multi-modal and multi-language support}, allowing image-based queries and responses in six languages including RTL scripts.
    \item \textbf{Resilient deployment} on a restricted hosting platform, with seed data fallback, connection failure caching, and a custom bootstrap pipeline.
    \item \textbf{Sub-20 ms} API response times for book data through aggressive caching and failure-state memorization.
\end{itemize}
 
\subsection{Future Roadmap}
 
Planned enhancements to the AI features include:
 
\begin{itemize}
    \item \textbf{Retrieval-Augmented Generation (RAG)}: Replace static \texttt{aiContext} prompts with a vector database (ChromaDB or Pinecone) of full-text book content. This would enable precise passage-level retrieval rather than relying on the LLM's training data, significantly improving factual accuracy and reducing hallucination risk.
 
    \item \textbf{Conversation persistence}: Store chat sessions in MongoDB to allow users to resume previous conversations and maintain history across browser sessions.
 
    \item \textbf{Streaming responses}: Implement Server-Sent Events (SSE) or WebSocket-based token streaming, reducing perceived latency from 2--4 seconds (full response wait) to under 500 ms (time-to-first-token).
 
    \item \textbf{Expanded provider support}: Add Anthropic Claude and OpenAI GPT-4 to the failover chain, with automatic latency-based routing.
 
    \item \textbf{Hosting upgrade}: Move the backend from the PythonAnywhere free plan to a \$5/month Developer plan (or an alternative like Railway/Render), enabling MongoDB connectivity, direct PyPI access, and elimination of the 30--40 second cold start.
\end{itemize}
 
\subsection{Summary}
 
Maududi's Legacy demonstrates how modern full-stack web development practices---dual-provider AI orchestration, aggressive caching, graceful degradation, and responsive multi-language UIs---can be applied to preserve and provide intelligent access to a significant scholarly corpus. 
 
From a senior developer's perspective, the project's defining achievement is the \textbf{AI reliability architecture}: the dual-provider failover (Gemini $\rightarrow$ Groq), accuracy enforcement via temperature control and prompt engineering, per-book AI personas with unique system prompts, structured citation extraction for verifiable answers, and the SWR + TTLCache caching strategy that keeps the user experience fast even on a restricted hosting platform. These patterns are directly applicable to any AI-powered content platform and represent production-grade AI integration done on a constrained budget.
 
The dual AI provider architecture (Gemini with Groq failover) provides both redundancy and capability breadth (native image handling on Gemini, specialized vision model on Groq) within a unified, low-latency chat interface --- and it cost nothing in API fees during development thanks to both providers' generous free tiers.
